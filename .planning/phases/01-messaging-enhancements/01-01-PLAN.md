Reference pattern — useGrid loadMore (src/hooks/useGrid.ts lines 63–78):
```typescript
const loadMore = useCallback(async () => {
  const { pages } = state
  if (!pages.length || !pages[pages.length - 1].has_more) return
  const nextPage = pages.length
  setState((s) => ({ ...s, loadingMore: true }))
  try {
    const next = await loadPage(nextPage, filters)
    setState((s) => ({ ...s, loadingMore: false, pages: [...s.pages, next] }))
  } catch (e) {
    setState((s) => ({ ...s, loadingMore: false, error: (e as Error).message ?? 'Network error' }))
  }
}, [state, filters, loadPage])
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `fetchMail(cursor)` URL construction | mitigate | Wrap cursor in `encodeURIComponent(cursor)` before appending to the URL so a malformed/poisoned cursor cannot break the query string or inject other params. Cursor flows server → server (response → next request), so authority remains server-side. |
| T-01-02 | Information Disclosure | useInbox accumulated threads in memory | accept | Threads already loaded in the SPA; pagination only extends what the user already has access to via their session cookie. No new data exposure. |
| T-01-03 | Denial of Service | User tap-spamming "Load more" | mitigate | `loadMore` guard returns early when `loadingMore === true`, preventing concurrent in-flight fetches. Button `disabled` attribute also blocks at the UI layer. |
| T-01-SC | Tampering | npm/pip/cargo installs | mitigate | No new dependencies added in this plan — `fetchMail`, `useInbox`, and the button reuse existing Preact + hand-rolled hooks/styles. No package-install task is present, so the legitimacy gate is N/A. |
